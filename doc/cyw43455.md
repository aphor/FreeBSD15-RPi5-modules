# CYW43455 SDIO WiFi — Reference

Cross-referenced summary of the SDIO host-driver-relevant behavior of the
Cypress/Infineon CYW43455 (FullMAC 802.11ac + BT5.0 combo), drawn from:

- **DS** = `_reference/infineon-cyw43455-datasheet-en.pdf`
- **L**  = Linux brcmfmac at `../raspbian_linux.git/drivers/net/wireless/broadcom/brcm80211/brcmfmac/{sdio,chip,bcdc}.c`
- **FB** = FreeBSD brcmfmac port at `../freebsd-brcmfmac.git/src/{sdio,sdpcm,fwil}.c` and `docs/03-sdio-reference.md`
- **T**  = facts established (or refuted) by our own testing in this driver
  (cited as **T:** with the symptom that established them)

The chip identifies as `chip_id=0x4345 rev=6`. Linux's chip table records
this combination as `BRCM_CC_43454_CHIP_ID` despite the part marking
"CYW43455" — branded as 43455 by Cypress/Infineon, the silicon is the
same as Broadcom's 43454 reference. SR detection at chip.c:1426 keys on
that ID.

---

## 1. Chip architecture (DS §1.2; L chip.c)

Three on-die clients share a BCMA backplane:

| Core            | Role                            | BCMA core ID |
|-----------------|---------------------------------|--------------|
| ARM Cortex-M4 (`CR4`) | Host CPU running the FullMAC firmware | `0x83e` |
| D11             | 802.11 MAC engine                | `0x812` |
| SDIO Device     | SDIO target controller, F1/F2 endpoint | `0x829` |
| ChipCommon      | PMU, OTP, GCI, sr_control        | `0x800` |
| ATCM/BTCM       | Tightly-coupled RAM for CR4      | (not BCMA cores) |

The CR4 runs out of TCM, which on this chip is **800 KB** total spread
across 8 banks (read from EROM at boot; see our `cyw_sdio.c` dmesg
"CR4 bank N" trace). The firmware blob `brcmfmac43455-sdio.bin`
(~609 KB) is downloaded to TCM base `0x00198000`.

NVRAM (~1.7 KB) is written to the **top** of TCM at
`ram_base + ram_size − nvram_len − 4`, with a trailing inverted-length
trailer that the firmware uses to find the blob. NVRAM contents pin
board-specific calibration; on Pi5 this is the `cyw43455-rpi.txt`
shipped with the firmware package.

## 2. SDIO function layout (DS §2; L sdio.c:200-)

| Func | Block | Role |
|------|-------|------|
| F0   | n/a   | Standard CCCR — `sdiob` programs this |
| F1   | 64    | Backplane access (registers, RAM, EROM) — synchronous CMD52/CMD53 |
| F2   | 512   | WLAN packet DMA (SDPCM frames in both directions) |
| F3   | —     | BT signaling on some SKUs; **not present** for our use |

Block sizes are configured via CCCR FBR registers (`sdiob` does this).
We use **F1 BS=64** the whole time and **F2 BS=64 during boot, then
F2 BS=512** after firmware handshake — Linux does the same; **T:**
trying to leave F2 at 64 post-boot causes BS-mismatch transfer
failures.

### F1 fixed-address FIFO at 0x10000

Backplane accesses use a sliding 32 KB window. The host writes a
"BAR0" address into F1 registers `0x1000A/B/C` (`SBADDRLOW/MID/HIGH`)
and then issues CMD52/CMD53 against F1 offsets `0x00000..0x07FFF` to
hit `BAR0 + offset`. SBADDR auto-clears on each access only for some
registers (`SBSDIO_SB_OFT_ADDR_*`); writes through it do **not**.

### F2 FIFO addresses

F2 transfers always target a fixed address (`addr_inc=0` in CMD53).
The conventional F2 FIFO base used by brcmfmac is `0x8000` (relative
to F2). Reads always return one full SDPCM frame; writes always
deliver one frame.

## 3. Backplane registers we actually touch (L sdio.c:62-; FB sdio.c)

All offsets are **F1** byte offsets unless otherwise noted.

| Symbol                        | Offset   | Notes |
|-------------------------------|----------|-------|
| `SBSDIO_FUNC1_SBADDRLOW`      | `0x1000A`| BAR0[15:8] |
| `SBSDIO_FUNC1_SBADDRMID`      | `0x1000B`| BAR0[23:16] |
| `SBSDIO_FUNC1_SBADDRHIGH`     | `0x1000C`| BAR0[31:24] |
| `SBSDIO_FUNC1_CHIPCLKCSR`     | `0x1000E`| Clock control (ALP_REQ, HT_REQ, ALP_AVAIL, HT_AVAIL, FORCE_HT) |
| `SBSDIO_FUNC1_SLEEPCSR`       | `0x1000E` *(typo? — actually `0x18000650` BP-relative)* | KSO + DEVON bits.  **T:** dunn dmesg shows reads of this at `0x1000e` succeeding — **the SDIOD-core SLEEPCSR is mirrored into the F1 register space at this same offset**. Linux brcmfmac calls this `SBSDIO_FUNC1_SLEEPCSR`, addressing it via F1 CMD52. |
| `SD_REG_INTSTATUS`            | `sdio_core + 0x020` | RW1C interrupt status (BP read32) |
| `SD_REG_HOSTINTMASK`          | `sdio_core + 0x024` | Bits that drive the SDIO IRQ line |

## 4. Clock state machine (DS §3.4; L sdio.c brcmf_sdio_clkctl)

`CHIPCLKCSR` (F1 reg `0x1000E`) carries:

```
   bit 7   FORCE_HT        force HT (160 MHz backplane) on
   bit 6   FORCE_ALP       force ALP (~26 MHz backplane) on
   bit 4   ALP_REQ         host requests ALP available
   bit 3   HT_REQ          host requests HT available
   bit 1   ALP_AVAIL       cleared/set by chip
   bit 0   HT_AVAIL        cleared/set by chip
```

State transitions exercised:

```
  POR ──────────────► [ALP off,HT off] ──set ALP_REQ──► poll ALP_AVAIL
                                                           │
                                                set HT_REQ │  (firmware download)
                                                           ▼
                                                  [ALP avail,HT pending]
                                                           │
                                                  poll HT_AVAIL
                                                           ▼
                                            [ALP+HT avail; FORCE_HT set]
                                                           │
                                         (firmware running)│
                                                           ▼
                                            ── detach: CHIPCLKCSR=0 ──► [off]
```

Backplane register access (BAR0 sliding-window reads) requires at
least **ALP** to be available. CR4 needs **HT**. We hold HT for the
whole firmware-download phase by setting `FORCE_HT` and clearing it
only just before passing control to the running firmware (which
manages its own clock requests).

**T:** During detach we re-enable ALP (via the same enable_clock path)
before issuing `cyw_arm_halt`'s backplane writes — without ALP the
SBADDR writes return EIO and the halt sequence is a no-op, which is
how `kldunload` + `kldload` started failing before the fix in commit
`708f92a`.

## 5. KSO — Keep SDIO On (L sdio.c:725- ; DS §4.3 "Sleep Modes")

The chip implements a backplane-sleep mechanism. While **KSO** is
cleared, the entire backplane is gated off; **every F1 register access
returns EIO**. F2 packet traffic still flows because that path has its
own clock.

`SLEEPCSR` (F1 reg `0x1000E`, same offset as `CHIPCLKCSR` — the
register decode is context-dependent on the SDIOD core; Linux defines
both at the same offset and they read back as distinct values
depending on which sub-decode the access hits):

```
   bit 1   KSO         Keep SDIO on
   bit 0   DEVON       Device on (read-only, reflects PMU state)
```

Wake sequence (Linux brcmf_sdio_kso_control, sdio.c:725):

1. Write `KSO=1` once — this is the wake-up command; **may not be
   ACKed** because the AOS (Always-On Subsystem) hasn't routed the
   write yet. The write is fire-and-forget.
2. Poll the register for `KSO=1 && DEVON=1`. The first **few** reads
   may return EIO or stale data; sleep `KSO_WAIT_US` (=50 µs) between
   tries, up to `PMU_MAX_TRANSITION_DLY/50 ≈ 110` tries.
3. Once `DEVON` reads back, the backplane is up.

Sleep sequence is just `KSO=0`, fire-and-forget. **No readback** —
the very write that clears KSO is the last thing the host can do
until something wakes it.

**Our driver** sets KSO once at probe (`KSO init: SLEEPCSR=0x03 [KSO
set]` in dmesg) and never sleeps the chip in normal operation
(Milestone 1/2 has no runtime sleep watchdog). Linux drives KSO from
its bus-state machine and the (firmware-side) "DSTANDBY" indications.

## 6. SDPCM framing (DS §2.4; L sdio.c, sdpcm.c)

Every F2 frame, in both directions, is wrapped in a fixed-layout
**SDPCM header** followed by optional protocol payload. Our struct
matches Linux's:

```
  offset 0    u16 len           total frame length, LE
  offset 2    u16 len_inv       ~len (used as a checksum)
  offset 4    u8  seq           8-bit sequence number, wraps
  offset 5    u8  chan_flags    low nibble = channel
  offset 6    u8  next_len      next-frame length hint (we ignore)
  offset 7    u8  data_offset   bytes from frame start to payload
  offset 8    u8  flow_ctl      device→host flow control bits
  offset 9    u8  credit        device→host: max seq the host may send
  offset 10   u8  reserved[2]
```

**Channels**:

| chan | name  | payload protocol             |
|------|-------|------------------------------|
| 0    | CTRL  | BCDC-encapsulated IOCTL/IOVAR |
| 1    | EVENT | BCDC event packets (assoc, scan results, …) |
| 2    | DATA  | 802.3 Ethernet frames |
| 3    | GLOM  | super-frame containing multiple sub-frames (we do not implement) |

**Credit flow control**: the chip tells the host its current "max
acceptable seq" via `hdr->credit`. The host tracks `tx_seq` (next seq
to send) and may only send while `(uint8_t)(rx_max - tx_seq) > 0`.
The seq number wraps modulo 256; the subtraction is intentionally
performed as `uint8_t` so it works across wrap. Seeding `rx_max = 4`
at attach is mandatory — without it the first TX never gets a credit
because the firmware never sends a starting credit update.

### BCDC (Broadcom DongleCmd) — CTRL/EVENT payload

```
  offset 0    u32 cmd     command code (e.g. WLC_GET_VAR=262)
  offset 4    u32 len     payload length following the BCDC header
  offset 8    u32 flags   bit 0: SET/GET, bits 16..31: transaction id
  offset 12   u32 status  device→host completion code
```

For an IOVAR, payload is the IOVAR name (NUL-terminated) followed by
SET data (or zero-filled for GET — the firmware fills it).

## 7. Interrupts and the INTSTATUS quirk (L sdio.c:200-, 2820-)

The SDIOD core advertises events to the host through `INTSTATUS`,
which is RW1C (write-1-to-clear). The host masks events with
`HOSTINTMASK` — bits set there drive the physical SDIO IRQ line.

### Bit layout (sdio.c:210-232)

| Bit   | Name              | Meaning |
|-------|-------------------|---------|
| 4     | `I_HMB_SW0` / `I_HMB_FC_STATE`  | flow-control state |
| 5     | `I_HMB_SW1` / `I_HMB_FC_CHANGE` | flow-control state changed |
| 6     | `I_HMB_SW2` / **`I_HMB_FRAME_IND`** | **firmware has a frame for host** |
| 7     | `I_HMB_SW3` / `I_HMB_HOST_INT`  | misc / host-defined |
| 4–7   | `I_HMB_SW_MASK`   | mask of all four mailbox bits |
| 23    | `I_XMTDATA_AVAIL` | "bits in fifo" — see quirk below |
| 29    | `I_CHIPACTIVE`    | chip transitioned doze→active |

Linux's `HOSTINTMASK = I_HMB_SW_MASK | I_CHIPACTIVE` (sdio.c:793).
**It does not include `I_XMTDATA_AVAIL`.** The Linux RX dispatcher
also explicitly `intstatus |= I_HMB_FRAME_IND;` in places where it
needs the next pass to read F2 — that bit is treated as
"there might be a frame", not a hardware guarantee.

### The reload INTSTATUS quirk (T:)

Cold-boot INTSTATUS right after fw handshake:
`0x20c000c0` = `I_HMB_HOST_INT | I_HMB_FRAME_IND | I_XMTDATA_AVAIL |
0x00400000 | I_CHIPACTIVE`.

After `cyw_arm_halt` + reload, INTSTATUS **at the moment the handshake
completes** (i.e. once `FWREADY` is set — see §7 TOHOSTMAILBOXDATA
below) varies: observed values include `0xa0400080` (`I_HMB_SW0` bit
set — within `I_HMB_SW_MASK`) and `0x20c000c0` (same as cold boot).
Either way, at least one `I_HMB_SW_MASK` or `I_XMTDATA_AVAIL` bit is
set by the time the firmware is ready.

The gate in `cyw_sdpcm_recv_one`:
```
if ((intstatus & (I_HMB_SW_MASK | I_XMTDATA_AVAIL)) == 0)
    return EAGAIN;
```
**must be kept**. Without it, the background callout issues a CMD53
read to an empty F2 FIFO every 50 ms. Those reads do not return a
clean all-zeros buffer — they return garbage that can partially pass
the `len`/`len_inv` check, corrupting the SDIO CAM queue state.
This was confirmed empirically: removing the gate (commit `b707160`)
caused a reproducible `camq_remove: out-of-bounds index -3` kernel
panic on both cold boot and reload. The gate was restored in commit
`d137875`.

**The gate is safe** as long as `cyw_sdpcm_attach()` — which starts the
callout — is only called after the FWREADY handshake. Because
FWREADY implies the gate bits are set (see above), the callout will
never see a false EAGAIN from the gate under normal attach ordering.

Linux does not need this gate because:

1. Its interrupt is **edge-triggered** off the SDIO IRQ line, not
   level-polled from INTSTATUS. The chip raises the IRQ when a
   response is ready; INTSTATUS is read only in the ISR/DPC, by which
   time the frame-available bits are set.
2. `brcmf_sdio_readframes()` also uses the SDPCM header `len`/`len_inv`
   pair as a secondary empty indicator after any IRQ-driven read, so
   a spurious read does not corrupt anything in the Linux path.

### Doorbell direction

Direction is from the chip's perspective:

- **TOSBMAILBOX** (`SMB_*`): host writes to wake/notify the chip.
  We write `SMB_HOST_INT (0x2)` after each F2 frame to nudge the
  firmware. Linux additionally writes `SMB_NAK / SMB_INT_ACK / …`
  bits during flow-control negotiation.
- **TOHOSTMAILBOXDATA**: chip writes to inform the host. The
  `HMB_DATA_*` bits encode firmware-up, NAK-handled, FWHALT, etc.

The fw-handshake value our driver prints is from `TOHOSTMAILBOXDATA`.
Relevant bits in the low nibble (T:):

| Constant             | Value  | Meaning |
|----------------------|--------|---------|
| `HMB_DATA_DEVREADY`  | `0x02` | SDPCM dispatcher started — set by the firmware immediately after ARM release, including on reload from TCM state. **This value is retained in the register across `cyw_arm_halt` and is still present when the next `kldload` starts downloading firmware.** |
| `HMB_DATA_FWREADY`   | `0x08` | WL init complete — firmware ready to accept IOVARs. Only written by the newly-downloaded firmware ~45 ms after ARM release. **This is the correct signal to wait for.** |

The upper nibble (`0x00040000`) is chip-revision-coded and opaque; we
do not mask or test it.

**Timing (T:):**

| Event                        | Cold boot | Reload (after `cyw_arm_halt`) |
|------------------------------|-----------|-------------------------------|
| `TOHOSTMAILBOXDATA` on entry (pre-download, ARM halted) | `0` | `0x00040002` (stale `DEVREADY` from previous firmware run) |
| After ARM release, t=0 ms   | `0x00040002` | `0x00040002` (still stale) |
| After ARM release, t=~45 ms | `0x00040008` (`FWREADY`) | `0x00040008` (`FWREADY`) |
| After ARM release, t=~65 ms | `0x00040008` (cold boot is slightly slower) | — |

The correct handshake waits for `TOHOSTMAILBOXDATA & HMB_DATA_FWREADY`
(commit `1b2f833`). The previous condition (`tohost != 0`) fired
immediately on the stale `DEVREADY` value during reload, causing the
`ver` IOVAR to be sent before the firmware was ready and therefore
always timing out (`firmware: unknown`).

## 8. ARM CR4 + D11 reset state machine (L chip.c:1052- and 1307-)

The host has fine-grained reset control over both cores via the BCMA
**RESET_CTL** and **IOCTL** registers, each at fixed BCMA offsets
relative to the per-core register window.

```
   ┌── disable_arm(CR4) ──┐         ┌── enable_arm(CR4) ──┐
   │ 1. ioctl |= CPUHALT  │         │ 1. ioctl = FGC|CLK   │
   │ 2. reset_ctl = RESET │   ──►   │ 2. reset_ctl = 0     │
   │ 3. (keep CPUHALT)    │         │ 3. ioctl = CLK       │
   └──────────────────────┘         │    (release halt)    │
                                    └──────────────────────┘
```

`set_passive` = `disable_arm(CR4)` + `coredisable(D11, PHYRESET|PHYCLOCKEN, PHYCLOCKEN)`.
The D11 sequence holds PHYRESET while clocked, then clears RESET_CTL,
then leaves only PHYCLOCKEN — D11 ends in a powered, clocked, idle
state.

This is **exactly** what `cyw_arm_halt` in our `cyw43455_sdio.c:233`
does, and is the equivalent of Linux's `brcmf_sdio_remove` →
`brcmf_chip_set_passive` → `brcmf_chip_cr4_set_passive`.

### What `set_passive` does NOT do (T:)

`set_passive` halts the running firmware but does **not**:

- Toggle WL_REG_ON / power-cycle the chip. Pi-class Linux installations
  rely on the platform's WL_REG_ON GPIO (defined in DT) to actually
  power-cycle the radio between module loads. **We have no WL_REG_ON
  control surface in our driver** — the Pi5 SDIO bridge does not
  expose a power-enable GPIO accessible from the sdiob bus.
- Reset PMU ChipControl 3 (where SR-capability lives). PMU CC3 is
  persistent across `set_passive`.
- Clear stale F2 FIFO contents.

This explains the observed cold-vs-reload differences (T:):

| Symptom                  | Cold boot              | After `cyw_arm_halt` reload |
|--------------------------|------------------------|------------------------------|
| `PMU CC3`                | `0xe1ab1f7c` (bit 2 set) | `0xe1b1f7c9` (bit 2 clear) |
| `SR capable`             | yes                    | no |
| `pre-download TOHOSTMAILBOXDATA` (ARM halted) | `0` | `0x00040002` (stale `DEVREADY`) |
| `fw handshake` time (waiting for `FWREADY`) | ~65 ms | ~45 ms |
| `INTSTATUS` at `FWREADY` | `0x20c000c0` (gate bits set) | `0xa0400080` or `0x20c000c0` (gate bits set) |

The PMU CC3 bit-2 (`PMU_CC3_SR_ENGINE_ENABLE`, chip.c:1430) is
**cleared by the running firmware during normal operation** — Linux
reads it before firmware download to decide whether to allocate the
SR scratch banks. After our halt+reload, the bit is still cleared
from the previous firmware's run; the chip is not power-cycled, so
PMU state persists.

Whether to "re-enable" SR by writing PMU CC3 from the host before
the next firmware download is **not done in Linux** (Linux relies on
WL_REG_ON for full reset). The remaining SR issue is **not blocking**
— the firmware runs correctly without SR on reload; `kldunload` /
`kldload` cycles work cleanly (5-cycle test verified, T:).

**What actually fixed reliable reload (T:):**

1. **Wait for `HMB_DATA_FWREADY` (bit 3)** in the post-ARM-release
   handshake (commit `1b2f833`).  `TOHOSTMAILBOXDATA` retains the
   previous firmware's `HMB_DATA_DEVREADY` (0x02) value across
   `cyw_arm_halt` — without this fix the handshake exited immediately
   on the stale value and the first IOVAR timed out.
2. **Keep the INTSTATUS gate** in `cyw_sdpcm_recv_one` (restored in
   commit `d137875`).  The gate is safe because INTSTATUS always has
   the data-available bits set by the time `FWREADY` is written
   (~45 ms after ARM release on reload, ~65 ms on cold boot).
   Removing the gate (as commit `b707160` briefly did) causes the
   50 ms callout to issue CMD53 reads to an empty F2 FIFO, corrupting
   the SDIO CAM queue and panicking the kernel
   (`camq_remove: out-of-bounds index -3`).

The SR-capability loss on reload remains an open cosmetic issue (see
§11 item 2) but does not affect driver correctness.

## 9. Per-IOVAR/IOCTL lifecycle (BCDC over SDPCM CTRL)

```
   host wants "ver" GET, txid=N:
   ┌────────────────────────────────────────────────────────────────┐
   │ wait for credit (rx_max − tx_seq > 0)                          │
   │ build [SDPCM hdr | BCDC hdr | "ver\0" | zero-pad]              │
   │ CMD53 F2 write → frame in firmware's RX FIFO                   │
   │ CMD52 backplane TOSBMAILBOX ← SMB_HOST_INT  (doorbell)         │
   └──────────────────────┬─────────────────────────────────────────┘
                          │
                  firmware processes, queues response
                          │
                          ▼
   ┌────────────────────────────────────────────────────────────────┐
   │ firmware: sets INTSTATUS bits (cold-boot only — see §7 quirk)  │
   │ firmware: writes response frame to F2 host-readable FIFO       │
   │ firmware: asserts SDIO IRQ line                                │
   └──────────────────────┬─────────────────────────────────────────┘
                          │
                  Linux: ISR → DPC → readframes
                  Ours:  50 ms callout → taskqueue thread → recv_one
                          │
                          ▼
   ┌────────────────────────────────────────────────────────────────┐
   │ CMD53 F2 read CYW_F2_BLKSIZE (=64) bytes                       │
   │ validate SDPCM hdr (len, ~len_inv, len ≥ HDR_LEN)              │
   │ if frame > BLKSIZE: CMD53 again for the tail                   │
   │ update rx_max from hdr->credit                                 │
   │ dispatch by hdr->chan_flags & 0xF:                             │
   │   CTRL: match BCDC flags.txid against ioctl_wait_id; cv_signal │
   │   EVENT: queue for net80211 event handler                      │
   │   DATA: ieee80211_input (in a future milestone)                │
   └────────────────────────────────────────────────────────────────┘
```

### Boot-time vs runtime path (our driver)

| Path        | `sc->sdpcm_running` | Caller     | RX side          |
|-------------|---------------------|------------|------------------|
| Boot-time   | `false`             | `cyw_fil_txrx` polls `cyw_sdpcm_recv_one` for up to 300 × 10 ms | Same thread |
| Runtime     | `true`              | `cyw_fil_txrx` writes frame, `cv_timedwait(3 s)` | Dedicated per-device taskqueue (commit `0ffb512`) |

**T:** The runtime path **must** run on a thread other than
`taskqueue_thread`. `sdiob` enqueues device discovery on
`taskqueue_thread` (sdiob.c:1050), and our `cv_timedwait` runs in
that thread; if the RX task is also on `taskqueue_thread`, the wait
self-starves and the IOCTL always times out. Fixed by allocating a
private `taskqueue_create("cyw43455_rx", …, taskqueue_thread_enqueue,
&sc->rx_tq)` with one dedicated thread.

## 10. Detach — what we have to do, ordered

Mirrors `brcmf_sdio_remove` (L sdio.c:4765). Confirmed working in
our `cyw_sdio_detach` (commit `708f92a`):

1. `cyw_cfg_detach` — `ieee80211_ifdetach` while net80211 still has
   the SDPCM task to back its `ic_transmit` (stub).
2. `cyw_sdpcm_detach` — clear `sdpcm_running`, `callout_drain`,
   `taskqueue_drain` + `taskqueue_free` the private RX queue. Wake
   any sleeping fwil caller with ENXIO so they don't hang on the way
   out.
3. `cyw_sdio_enable_clock(sc)` + `DELAY(20000)` — ensure ALP for the
   passive-reset writes and give the chip a Linux-equivalent settle
   window.
4. `cyw_arm_halt(sc)` — `set_passive`: halt CR4, reset D11.
5. `CHIPCLKCSR=0` — release the host's HT/ALP request.
6. `sdio_disable_func(F2)`; `sdio_disable_func(F1)`.

Steps **3 → 4 → 5 → 6 ordering is required**. If the SDIO functions
are disabled before the ARM is halted, the running firmware will (a)
clear KSO and (b) modify PMU CC3 such that the next attach finds the
chip in the "reload" state described in §8.

## 11. Open issues and unknowns

These are explicitly **not** yet confirmed against documentation or
tested:

1. **F2 FIFO flush on detach** — we don't currently drain F2 before
   `disable_func(F2)`. If responses to in-flight IOVARs are pending,
   they may be left in the chip's FIFO and reappear on the next
   attach. (**Not** a contributor to the CAM panic seen under repeated
   reload — that panic was caused by the background callout issuing
   CMD53 reads to an empty F2 FIFO after the INTSTATUS gate was
   wrongly removed; see T: note in §7.  Whether stale FIFO contents
   cause any observable problem in practice remains untested.)
2. **PMU CC3 restore for SR** — speculation that writing back the
   cold-boot CC3 value before firmware download would restore SR
   capability on reload. Not attempted; no documentation found.
3. **TOSBMAILBOX bit semantics** — Linux uses `SMB_INT_ACK`, `SMB_NAK`,
   `SMB_USE_OOB`, `SMB_DEV_INT` and `SMB_HOST_INT`. Our code only
   writes `SMB_HOST_INT`. Whether the firmware actually needs any of
   the other bits for our use case has not been tested.
4. **Why `INTSTATUS` bit 22 is set in both cold-boot and reload**
   (`0x00400000`). This bit is not named in Linux's headers; it
   appears to be reserved. We do not act on it.
5. **TOHOSTMAILBOXDATA layout above the low nibble** — the values
   `0x00040008` vs `0x00040002` differ in bits 0-3 only. The 0x40000
   constant component is unexplained.
6. **~~`fw handshake at 0 ms` on reload~~** — **RESOLVED** (commit
   `1b2f833`).  Root cause: `TOHOSTMAILBOXDATA` retains the previous
   firmware's `HMB_DATA_DEVREADY` value (0x02, bit 1) across
   `cyw_arm_halt`.  The old handshake polled for `tohost != 0` so it
   exited immediately on the stale value, before the newly-downloaded
   firmware had initialised.  Fix: poll for `HMB_DATA_FWREADY`
   (bit 3, 0x08) specifically — this bit is only written by newly-
   downloaded firmware ~45 ms after ARM release.  The symptom
   (`fw handshake at 0 ms` → first IOVAR timeout → `firmware:
   unknown`) is no longer reproducible; every reload now shows
   `fw handshake at ~45 ms` and correct firmware version.

## 13. Linux firmware command sequence from boot to first scan

Source files: `brcmfmac/common.c` (`brcmf_c_preinit_dcmds`),
`brcmfmac/core.c` (`brcmf_bus_started`, `brcmf_netdev_open`),
`brcmfmac/cfg80211.c` (`brcmf_config_dongle`, `brcmf_cfg80211_scan`,
`brcmf_do_escan`, `brcmf_cfg80211_change_iface`).

### Phase 1 — SDIO bus attach (no firmware commands yet)

Linux's `brcmf_sdio_probe` downloads firmware, releases ARM, waits for
`HMB_DATA_FWREADY`, enables F2, then calls `brcmf_bus_started`.

| Step | Action | Firmware state after |
|------|--------|----------------------|
| 1 | ARM release (IOCTL cleared, RESET_CTL=0) | Firmware booting, BSS not yet configured |
| 2 | Wait for `HMB_DATA_FWREADY` (~45–65 ms) | Firmware running, SDPCM/BCDC stack ready |
| 3 | F2 enabled, `HOSTINTMASK` written | F2 RX path live; firmware queues E_IF ADD |
| 4 | `brcmf_bus_change_state(BRCMF_BUS_UP)` | Host flag only — no firmware command |

### Phase 2 — `brcmf_c_preinit_dcmds` (common.c:268)

Called from `brcmf_bus_started` after F2 is ready. **No WLC_DOWN, no
WLC_UP, no WLC_SET_INFRA issued here.** The BSS is left in whatever
initial state the firmware chose after boot.

| Step | Command / IOVAR | Value | Notes |
|------|-----------------|-------|-------|
| 1 | `cur_etheraddr` GET | — | Read factory MAC |
| 2 | `C_GET_REVINFO` (cmd 318) | — | Read chip/firmware revision |
| 3 | CLM blob download | — | `brcmf_c_process_clm_blob` via IOVAR; Pi 5 ships without CLM blob — skipped |
| 4 | `ver` GET | — | Read firmware version string |
| 5 | `mpc` SET | **1** (enabled) | MPC on by default; disabled only right before escan |
| 6 | `event_msgs` GET + SET | add `E_IF` bit | Subscribe to firmware events |

**Observation:** Linux sets `mpc=1` at init, and firmware starts in its
own DEFAULT state.  Linux does **not** issue `WLC_DOWN`, `WLC_SET_INFRA`,
`btc_mode`, `allmulti`, `roam_off`, or any form of `pm` here.

### Phase 3 — `brcmf_cfg80211_attach` / `brcmf_net_attach`

Pure host-side setup (cfg80211/wiphy registration, net_device ops, no
firmware commands). `brcmf_netdev_open` is registered as `ndo_open`.

### Phase 4 — First `ifconfig wlan0 up` → `brcmf_netdev_open`

`brcmf_netdev_open` → `brcmf_cfg80211_up` → `__brcmf_cfg80211_up`
(cfg80211.c:8019):

```
set_bit(BRCMF_VIF_STATUS_READY, &ifp->vif->sme_state)  ← host-only flag
brcmf_config_dongle(cfg)                                ← firmware sequence below
```

`brcmf_config_dongle` (cfg80211.c:7968), in order:

| Step | Command / IOVAR | Value | Source | Notes |
|------|-----------------|-------|--------|-------|
| 1 | `BRCMF_C_UP` (cmd 2) | 0 | cfg80211.c:7985 | Bring BSS up. Arg is 0 (ignored by firmware). |
| 2 | `BRCMF_C_SET_SCAN_CHANNEL_TIME` (cmd 185) | 40 ms | `brcmf_dongle_scantime` | Active dwell per channel |
| 3 | `BRCMF_C_SET_SCAN_UNASSOC_TIME` (cmd 187) | 40 ms | `brcmf_dongle_scantime` | Unassoc dwell |
| 4 | `BRCMF_C_SET_SCAN_PASSIVE_TIME` (cmd 258) | 120 ms | `brcmf_dongle_scantime` | Passive dwell |
| 5 | `BRCMF_C_SET_PM` (cmd 86) | 0 (PM_OFF) or 2 (PM_FAST) | cfg80211.c:7989 | Depends on `cfg->pwr_save`; Pi 5 default: PM_OFF=0 |
| 6 | `bcn_timeout` IOVAR | 4 (roam off) or 8 (roam on) | `brcmf_dongle_roam` | Beacon loss timeout |
| 7 | `roam_off` IOVAR | 1 (disabled) | `brcmf_dongle_roam` | Suppress internal roam |
| 8 | `BRCMF_C_SET_ROAM_TRIGGER` (cmd 55) | `[-75 dBm, BRCM_BAND_ALL]` | `brcmf_dongle_roam` | Two `__le32` words |
| 9 | `BRCMF_C_SET_ROAM_DELTA` (cmd 57) | `[20, BRCM_BAND_ALL]` | `brcmf_dongle_roam` | Two `__le32` words |
| 10 | `BRCMF_C_SET_INFRA` (cmd 20) | 1 (infra/STA) | `brcmf_cfg80211_change_iface` | Called only for non-AP modes; not called if `iftype==AP` |
| 11 | arp/nd offload IOVARs | — | `brcmf_configure_arp_nd_offload` | `arpoe`, `arp_ol`, `ndoe` — offload to firmware |
| 12 | `BRCMF_C_SET_FAKEFRAG` (cmd 219) | 1 | cfg80211.c:8007 | Frameburst / fake fragmentation |
| — | `cfg->dongle_up = true` | — | — | Guards single-run; subsequent ifconfig up/down cycles skip this block |

**Critical note on WLC_DOWN:** Linux **never issues WLC_DOWN** in the
preinit or config_dongle paths for a STA interface.  `BRCMF_C_DOWN`
only appears in the AP-mode setup code (`brcmf_cfg80211_start_ap`,
cfg80211.c:5218/5394) and the P2P path (p2p.c:434). Issuing WLC_DOWN
at attach time is not part of the Linux pattern for STA operation.

**Critical note on WLC_SET_INFRA ordering:** `BRCMF_C_SET_INFRA` is
issued **after** `BRCMF_C_UP`, inside `brcmf_cfg80211_change_iface`.
Linux never issues SET_INFRA before C_UP; the order is
C_UP → scan timing → SET_PM → roam → SET_INFRA → fakefrag.

### Phase 5 — `ifconfig wlan0 scan` → `brcmf_cfg80211_scan` (cfg80211.c:1516)

| Step | Action | Notes |
|------|--------|-------|
| 1 | `check_vif_up(vif)` | Returns false (→ -EIO) if `VIF_STATUS_READY` not set; this flag was set in Phase 4 step 1 |
| 2 | Check `SCAN_STATUS_BUSY/ABORT/SUPPRESS` | Abort if scan already in progress |
| 3 | `brcmf_do_escan(vif->ifp, request)` | Calls `brcmf_scan_config_mpc(ifp, 0)` **first** |
| 4 | `brcmf_scan_config_mpc(ifp, 0)` | Sets `mpc=0` (radio stays awake) — only if `NEED_MPC` quirk is set for this chip |
| 5 | `escan` IOVAR SET | Firmware-driven scan; results delivered as `E_ESCAN_RESULT` events |
| 6 | On completion or error: `brcmf_scan_config_mpc(ifp, 1)` | Restore mpc=1 |

**`NEED_MPC` quirk:** Checked via `brcmf_feat_is_quirk_enabled(ifp,
BRCMF_FEAT_QUIRK_NEED_MPC)`.  This is set for chips that need MPC
toggled around scans.  If the quirk is not set, `mpc` is never touched
by the scan path.  For the CYW43455 (43454 silicon), check whether this
quirk applies.

### Comparison: Linux vs. cyw43455 driver (current state)

| Phase | Linux | cyw43455 (current) | Delta |
|-------|-------|--------------------|-------|
| Attach: WLC_DOWN | ✗ never | ✓ issued | **Extra; not in Linux** |
| Attach: WLC_SET_INFRA | ✗ never | ✓ issued | **Extra; not in Linux** |
| Attach: mpc | mpc=1 | mpc=0 | **Inverted default** |
| Attach: btc_mode | ✗ never | btc_mode=0 | Extra; harmless? |
| Attach: roam_off | ✗ never | roam_off=1 | Extra (Linux does it in config_dongle) |
| Attach: allmulti | ✓ (in netdev context) | ✓ | Same intent |
| Attach: WLC_UP | ✗ never | ✓ (after sdpcm_attach) | **Linux defers to ifconfig up** |
| ifconfig up: WLC_UP | ✓ config_dongle step 1 | ✗ (already done in attach) | Sequencing inverted |
| ifconfig up: SET_INFRA | ✓ via change_iface, after C_UP | ✓ (in #if 0) | Linux issues after C_UP |
| Scan: mpc=0 before escan | ✓ (brcmf_do_escan) | mpc=0 set at attach (stays 0) | Linux toggles; ours is always 0 |

**Suspected root cause of BCME_NOTUP (T: unresolved):**

Two candidates, ranked by strength:

1. **WLC_DOWN at attach time** — Linux never issues WLC_DOWN for STA
   mode during attach.  Issuing it may push the BSS into a DOWN state
   from which `WLC_UP` issued after `cyw_sdpcm_attach` cannot recover it
   (the condvar-path WLC_UP runs while the RX task is already servicing
   frames, so the E_IF events from the DOWN/UP cycle may not be cleanly
   serialized against subsequent IOVARs).

2. **WLC_UP placement** — Linux issues WLC_UP from `brcmf_config_dongle`
   on first `ifconfig up`, not from attach.  The firmware may require that
   WLC_UP arrive after the host network stack has committed to STA mode
   (i.e., after `brcmf_net_attach`/`ieee80211_ifattach` have registered
   the interface), not during attach.  Our driver calls it in a window
   where the interface is not yet visible to the OS.

## 12. Reading list, in priority order

When stuck, read these — in this order — before guessing:

1. `../raspbian_linux.git/drivers/net/wireless/broadcom/brcm80211/brcmfmac/sdio.c`
   sections: ~lines 200–280 (constants), 700–800 (KSO/HOSTINTMASK),
   1100–1170 (hostmail), 2580–2880 (DPC main loop), 3800–3900 (ISR),
   4400–4500 (probe), 4750–4800 (remove).
2. `../raspbian_linux.git/drivers/net/wireless/broadcom/brcm80211/brcmfmac/chip.c`
   sections: 1050–1100 (`brcmf_chip_disable_arm`), 1280–1320
   (`brcmf_chip_cr4_set_passive`), 1400–1480 (`brcmf_chip_sr_capable`),
   1500–1570 (`brcmf_chip_set_active`).
3. `../raspbian_linux.git/drivers/net/wireless/broadcom/brcm80211/brcmfmac/bcdc.c`
   — BCDC framing, especially `brcmf_proto_bcdc_msg`.
4. `../freebsd-brcmfmac.git/src/sdio.c` and `docs/03-sdio-reference.md`,
   `docs/12-sdio-retro.md` — a parallel FreeBSD attempt with a more
   complete RX path. Most useful for net80211 wiring; the SDIO
   bring-up has different bugs than ours.
5. `_reference/infineon-cyw43455-datasheet-en.pdf` —
   §1 chip block diagram; §2.4 SDIO host interface; §3 power/clocks;
   §4.3 sleep modes (KSO); §6 register summary. The datasheet is
   silent on SDPCM/BCDC framing — those are software conventions
   shared across the brcmfmac chip family.
